# Improvement Plan 1

## Analysis Summary

The initial build and run verification shows the app launches correctly with the Onboarding screen. Core functionality is in place. The following improvements are needed based on deep analysis.

## Improvements

### 1. Fix Core Data Model Integration
- **Need**: The Core Data model file (Lullab.xcdatamodeld) needs to be properly recognized by Xcode. The current model uses manual NSManagedObject subclasses which is correct, but we need to ensure the model is in the project's build sources.
- **Generation Rules**: Verify the xcdatamodeld is in the project's build phase. If not, add it via the pbxproj. The model should have Codegen = Manual/None since we have custom subclasses.
- **Verification**: Build succeeds with Core Data model loaded. App can create and fetch CDEvent and CDBaby entities without crash.

### 2. Add Info.plist Network Security Configuration
- **Need**: Contact Support requires outbound HTTPS network access. Need NSAppTransportSecurity configuration in Info.plist.
- **Generation Rules**: Add NSAppTransportSecurity with NSAllowsArbitraryLoads = false to Info.plist. This allows HTTPS by default (which is what we need for the feedback backend).
- **Verification**: Feedback submission works without network errors.

### 3. Improve Onboarding Flow - Add Keyboard Dismissal
- **Need**: When user types baby name in BabySetupView, the keyboard may cover the Save button. Need a way to dismiss keyboard.
- **Generation Rules**: Add `.scrollDismissesKeyboard(.interactively)` to the ScrollView in BabySetupView. Also add a tap gesture to dismiss keyboard.
- **Verification**: User can tap outside text field to dismiss keyboard and see the Save button.

### 4. Add Empty State for Timeline
- **Need**: Timeline shows "No events yet" text but could be more visually engaging with an illustration.
- **Generation Rules**: Keep current simple text for now. The empty state is functional and clear.
- **Verification**: Empty state displays correctly when no events exist.

### 5. Ensure Dark Mode Consistency
- **Need**: All views must render correctly in dark mode (our primary design). Some system components (DatePicker, List) may need explicit styling.
- **Generation Rules**: Verify all views use Color.lullabXxx tokens. System components like DatePicker should inherit the dark theme automatically.
- **Verification**: All views render with dark backgrounds and light text in dark mode.

## Items NOT Improved (Deferred)

- WidgetKit extension: Requires separate target, will be added in future iteration
- Live Activity: Requires Info.plist entry (NSSupportsLiveActivities = YES), will add manually
- CloudKit: Requires iCloud capability in Apple Developer portal, will configure manually
- Doctor Report PDF generation: Complex feature, deferred to next iteration
- WHO Growth Charts: Requires chart data, deferred to next iteration
